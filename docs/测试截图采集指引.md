# 测试截图采集指引

本文用于配合 [项目测试文档](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/docs/第十九届全国大学生软件创新大赛-项目测试文档.md) 完成测试结果截图采集。所有截图建议统一保存到 `docs/test-images/` 目录，文件名与文档图号保持一致。

## 一、截图准备

### 1. 环境准备

建议按以下顺序准备：

1. 前端界面截图前执行 `npm run dev`。
2. 系统测试截图前执行 `npm run test:system`。
3. 接口与压测截图前执行 `node scripts/data-server-api-perf-test.mjs`。
4. 构建截图前执行 `npm run build`。

### 2. 截图建议

1. 当前设备为 macOS，可使用 `Shift + Command + 4` 进行区域截图。
2. 界面截图优先保留完整窗口，不建议只截局部控件。
3. 终端截图建议放大终端字号，确保 `passed`、`avgMs`、`throughputRps` 等关键字清晰可见。
4. 若一个结论需要体现前后状态变化，建议使用拼图工具做成一张横向组合图。

## 二、逐图截图任务

### 图 3-1 家属端与老人端视图切换结果图

目标：证明主界面可以在“老人端(App)”和“家属端(后台)”之间切换，且内容区域同步变化。

操作步骤：

1. 执行 `npm run dev` 并打开页面。
2. 保持顶部导航栏可见。
3. 先点击 `老人端 (App)`，再点击 `家属端 (后台)`。
4. 选择更能体现差异的一个时刻截图，或制作左右拼图。

截图验收点：

1. 顶部出现 `老人端 (App)` 与 `家属端 (后台)` 按钮。
2. 当前选中项存在明显高亮。
3. 主内容区域能看出与按钮状态对应的页面内容。

### 图 3-2 家属端触发用药提醒后老人端反馈结果图

目标：证明家属操作能直接驱动老人端提醒弹层。

操作步骤：

1. 在前端界面进入老人端主视图，并确保右侧可见“家属联动控制台”。
2. 点击按钮 `打开用药引导`。
3. 等待老人端出现药物提醒弹层。
4. 截图时保留控制台按钮和老人端弹层。

截图验收点：

1. 画面中可看到 `打开用药引导` 按钮。
2. 老人端弹层中应出现药品名称、剂量、用途或服用说明。
3. 最好保留“我已服药”按钮，证明后续流程可继续。

### 图 3-3 已服药确认结果图

目标：证明提醒流程有明确完成态。

操作步骤：

1. 在图 3-2 的基础上继续。
2. 点击老人端弹层按钮 `我已服药`，或在联动控制台点击 `✅ 已服药确认`。
3. 在绿色确认态“已记录服药”出现时截图。
4. 若条件允许，建议拼接“提醒态 + 确认态”两张图。

截图验收点：

1. 画面中出现 `已记录服药` 文案。
2. 最好同时能看出这是由前一阶段提醒流程演变而来。

### 图 3-4 走失提醒弹窗结果图

目标：证明系统能够在走失场景中主动弹窗并提供帮助选项。

操作步骤：

1. 在家属端页面找到模拟按钮 `⚠️ 模拟: 疑似走失 (12h)`，或通过相应侧边栏入口触发走失场景。
2. 等待老人端出现走失提醒弹窗。
3. 截图时保留弹窗主体。

截图验收点：

1. 标题中应出现 `已离开安全区域`、`检测到可能迷路` 等文字之一。
2. 应出现 `帮我回家`、`联系家人` 按钮。
3. 最好保留倒计时提示，如 `xx秒后将自动联系家人`。

### 图 3-5 认知健康报告结果图

目标：证明系统可以生成并展示认知可视化报告。

操作步骤：

1. 在老人端界面或家属联动控制台中触发认知场景。
2. 推荐优先尝试 `S7 认知评估+健康日报` 场景按钮，或按钮 `🧠 模拟认知问答记录`。
3. 当 `认知健康报告` 弹窗打开后截图。

截图验收点：

1. 标题必须包含 `认知健康报告`。
2. 画面中至少包含以下元素之一：综合评分、五维度评分、近 7 天趋势、异常提醒。
3. 若能同时保留 `今日详情` / `趋势分析` 页签更好。

### 图 3-6 语音交互时延测试结果图

目标：为语音响应性能提供量化证据。

操作步骤：

1. 执行 `npm run test:functional`。
2. 在终端输出中找到以下关键日志：
   `recognition_latency_ms=40`
   `reply_latency_ms=55`
   `clone_ready_latency_ms=80`
3. 截图时保留命令行和这些数值。

截图验收点：

1. 能看到 `npm run test:functional` 的上下文。
2. 三个时延数字至少保留两个，最好三个都完整显示。

### 图 4-1 系统集成测试通过结果图

目标：证明 Bridge 与 Data Backend 的系统级链路真实通过。

操作步骤：

1. 执行 `npm run test:system`。
2. 在终端中停留在全部通过的结果界面。
3. 尽量保留部分关键服务日志。

截图验收点：

1. 应显示 `3 passed` 或 `14 passed`。
2. 最好能看到 `tests/system/bridge-data-backend.test.ts`。
3. 最好能看到 `POST /api/elder/events -> 200`、`GET /healthz -> 200` 等日志。

### 图 4-2 接口测试结果图

目标：证明真实接口调用成功。

操作步骤：

1. 执行 `node scripts/data-server-api-perf-test.mjs`。
2. 截图时保留 JSON 输出中 `apiTests` 部分。

截图验收点：

1. 应包含 `GET /healthz`、`GET /api/elder?elderId=elder_demo`、`POST /api/elder/events`。
2. 对应测试项应显示 `passed: true`。

### 图 4-3 压力测试结果图

目标：证明接口在并发下仍稳定，并且事件不丢失。

操作步骤：

1. 继续使用 `node scripts/data-server-api-perf-test.mjs` 的终端结果。
2. 截图聚焦 `loadTests` 段落。

截图验收点：

1. 应出现 `successRate: 100`。
2. 应出现 `avgMs`、`p95Ms`、`throughputRps`。
3. 对事件压测应保留 `actualEventCount: 100`。

### 图 4-4 生产构建结果图

目标：证明项目已具备可构建性，同时保留包体风险信息。

操作步骤：

1. 执行 `npm run build`。
2. 在构建完成后截图。

截图验收点：

1. 应出现 `built in 2.28s` 或当前实际构建耗时。
2. 应出现 `dist/index.html` 和主 JS 产物大小。
3. 应保留 chunk size warning。

## 三、截图完成后的回填方式

完成截图后，建议按以下方式回填：

1. 将图片保存到 `docs/test-images/`。
2. 打开测试文档中对应的“截图占位”区域。
3. 用真实图片替换占位说明，或在占位区下方直接插入图片。
4. 若图片文件名与本指引不一致，同步修改文档中的文件名说明，避免后续整理混乱。
