## 1 测试计划

### 1.1 测试策略与目标

本项目为面向老年陪护场景的 AI 养老照护系统，核心能力包括健康监测、走失预警、用药提醒、认知辅助、家属联动以及 OpenClaw Bridge 数据同步。项目存在“前端交互 + 本地数据服务 + 桥接同步 + 外部语音能力”的多层链路，因此本轮测试采用分层测试策略，并围绕关键业务链路的正确性、稳定性和可追溯性组织测试活动。

本项目业务链路涉及老人端、家属端、本地数据服务、Bridge 同步服务以及外部语音能力，具有多角色交互、事件驱动和状态更新频繁等特点。测试重点包括核心状态数据一致性、提醒与告警链路完整性、异常输入容错处理、接口并发写入稳定性以及真实语音服务接入后的可用性。测试难点主要集中在跨模块协同验证、异步事件时序控制、数据持久化正确性以及外部服务引入后的时延波动控制。因此，本轮测试采用分层测试策略，对底层逻辑、界面流程、系统集成、接口行为、并发稳定性和构建交付能力进行分别验证。

在测试策略设计阶段，本项目将“业务风险优先级”作为测试资源分配的首要依据。具体而言，凡是直接影响老人提醒、家属联动、异常告警、风险记录和后续回放复核的链路，均优先纳入本轮验证范围；凡是仅影响局部显示效果、非关键辅助信息或后续可替换实现的内容，则根据时间与环境条件进行阶段性安排。该策略有助于确保有限测试时间优先覆盖高风险、高频使用和高可见度功能，从而提升测试活动对当前版本质量判断的有效性。

本轮测试还遵循“先稳定底层，再验证上层；先验证内部闭环，再验证外部依赖”的组织原则。底层数据结构、服务状态流转、异常映射逻辑通过单元测试完成早期校验；用户可见的交互流程通过功能测试验证；跨模块契约和真实 HTTP 读写行为通过系统测试和接口测试确认；对外部语音服务则采用独立 live 联调方式单独执行。该组织方式的目的在于降低问题定位成本，确保一旦发生异常，可快速判断问题位于逻辑层、界面层、集成层还是外部依赖层。基于这一原则，本轮测试被组织为六个相互衔接的验证层次：单元测试侧重验证状态归一化、并发写入、服务事件发布、告警判定和消息防误投等底层逻辑；功能测试侧重验证老人端、家属端、认知报告、走失提醒、用药提醒和语音交互等用户可见流程；系统测试侧重验证 Bridge 与 Data Backend 的真实 HTTP 读写回路、契约头部、节流去重和跨模块 happy path；接口测试侧重验证健康检查、老人数据查询、状态写入、事件写入、媒体上传和异常返回；压力测试侧重评估关键接口在本地回环环境下的响应时延、吞吐量与数据完整性；构建验证则用于确认项目当前版本具备稳定完成打包交付的能力。通过这种分层划分，测试结果可以分别回答“底层是否正确、流程是否成立、集成是否贯通、接口是否稳定、并发是否可靠、产物是否可交付”等不同质量问题。

质量等级评定不单纯依据测试通过数量，而是综合考量测试覆盖对象的重要程度、关键链路是否贯通、异常路径是否得到验证、性能是否达到当前使用场景要求，以及未覆盖范围是否已被明确识别和说明。基于本轮结果，项目已经具备作为竞赛验收版本进行展示、答辩说明、联调和阶段交付的基础条件；同时，文档中也保留了对语音克隆时延、包体体积和真机验证空白等问题的风险说明，以避免质量结论与实际工程状态不一致。

### 1.2 测试范围

本轮测试范围覆盖项目的主要业务链路以及后端数据主链路。

测试范围的确定遵循业务重要性、风险等级和可验证性原则，优先覆盖家属端联动、老人端提醒弹层、认知报告、走失告警、数据落盘、接口并发稳定性等核心场景，以保证测试结果能够对应系统主要功能和关键风险点。

在测试广度方面，本轮测试已覆盖前端主要交互视图、核心业务服务、数据服务主接口、Bridge 契约、并发写入路径以及构建发布流程，能够反映当前系统在“输入、处理、存储、反馈、交付”五个基本环节中的整体质量状态。在测试深度方面，文档不仅覆盖正常输入下的功能验证，还补充了非法 section、非法上传、服务未配置、服务不可用、并发写入、长时延语音推理等异常或边界情况，以便对系统的稳定性边界形成更完整判断。

需要强调的是，本轮测试范围并非追求形式上的“全覆盖”，而是强调“对关键功能与关键风险形成有效覆盖”。例如，家属端与老人端双端联动虽然只对应少量页面，但其背后涉及状态流转、消息触发、界面反馈和后续收口，是系统整体可用性的重要体现；又如，`POST /api/elder/events` 虽然只是单个接口，但它承载了异常事件记录、状态追踪和后续回放等多个业务责任，因此在测试设计中获得了更高权重。


| 维度        | 纳入测试范围的内容                                |
| --------- | ---------------------------------------- |
| 前端交互      | 家属端/老人端视图切换、异常状态展示、认知报告、走失提醒、用药提醒、家属联动按钮 |
| 业务服务      | 黄昏综合征风险评估、健康阈值判断、走失事件模拟、用药提醒状态流转、消息防误投   |
| 数据服务      | 老人档案读取、状态写入、事件写入、媒体上传、异常 section 返回      |
| Bridge 契约 | Token 头部、状态去重、事件同步、家属通知动作                |
| 性能稳定性     | 本地回环环境下接口响应时间、并发吞吐与数据完整性                 |
| 发布质量      | 前端生产构建可执行性                               |


从测试深度看，本轮优先覆盖业务风险较高的核心链路，重点围绕老人状态数据是否能够被正确写入、回读并补齐默认结构、并发场景下事件与 section 更新是否会互相覆盖或丢失、家属端发起的提醒与异常动作是否能够正确反馈到老人端界面、Bridge 请求是否满足 header 与 body 契约以避免接入 OpenClaw 时出现协议不兼容，以及核心数据服务在中等并发下是否仍然保持 100% 成功率和数据完整性等问题展开。这样安排的目的，是使测试结论不仅能够回答“系统是否能运行”，还能够进一步回答“系统在高风险业务环节中是否保持稳定、可解释和可追溯”。

本轮未纳入测试范围或仅部分覆盖的内容如下：


| 未完全覆盖项                     | 原因说明                                                             |
| -------------------------- | ---------------------------------------------------------------- |
| 真正的语音识别 / TTS / 语音克隆在线服务联调 | 已在独立 live 环境补测通过；但当前仅验证本机 WebSocket 在线服务，未覆盖公网云服务 SLA 与移动端真机音频链路 |
| Android 原生后端、穿戴设备、传感器联调    | 当前测试环境为本机 Node/Vitest 环境，未接入真实设备或模拟器                             |
| 长时稳定性测试（8h/24h endurance）  | 本轮重点为功能与接口稳定性验证，未进行长时间 soak test                                 |
| 外部通知渠道（飞书、短信等）真实发送         | 系统测试验证了契约，不包含真实第三方网关调用                                           |


未完全覆盖项主要受测试环境、设备条件和执行成本限制。为兼顾日常回归效率与联调准确性，真实语音服务采用独立 live 测试入口执行；常规回归用于验证主要功能流程，独立 live 联调用于验证外部依赖链路。这种分层方式有利于后续持续维护。

对于未完全覆盖项，文档采取明确记录边界的方式进行说明，目的是防止测试结论被误解为对所有运行环境和所有外部依赖均已完成同等深度验证。例如，本轮已完成本机 WebSocket 语音服务联调，但这并不等同于已经完成公网环境、复杂网络环境、不同终端硬件条件下的同级别验证；又如，系统已具备 Bridge 契约验证能力，但这并不等同于第三方通知网关已在真实生产渠道中完成端到端发送验证。通过这种边界说明，可以使文档结论保持客观、可审计和可追溯。

建议后续继续从边界值和异常恢复两个方向深化验证。对于黄昏综合征和健康评估模块，应增加更多阈值边缘用例，重点观察分数恰好处于临界区间时的行为是否稳定、解释是否一致；对于 `openclawSyncService`，应补充请求超时、重试策略和异常恢复场景，验证在外部同步环境不稳定时系统能否保持可控状态；对于用药与走失模块，则建议增加连续触发和重复事件去重测试，以避免真实设备重复上报时前端出现弹窗抖动、状态频繁覆盖或用户感知混乱等问题。

#### 2.2.3 测试经验总结

养老场景下的业务服务测试不能只验证“有没有结果”，还需要进一步验证“错误结果是否会伤害用户体验”以及“异常情况下系统是否会以可控方式失败”。本项目将家属消息误投、Bridge 未配置和重复状态同步等场景纳入服务层测试，是有现实必要性的，因为这些问题一旦出现在真实照护场景中，往往比单纯的功能失效更容易造成用户误解。

本节测试结果表明，相关业务能力已经能够通过明确的服务逻辑和事件输出进行验证，这不仅提高了规则本身的可解释性，也为界面层、系统层和后续联调层提供了可以复用的稳定基础。后续若增加更多评估规则或告警场景，仍建议延续当前“先验证服务规则、再验证界面表现”的测试组织方式。

就认知与异常提醒功能而言，测试结果同样具有较好的解释性。走失提醒、认知报告等功能在组件级测试中均能完成展示、交互和关闭流程，说明异常场景处理与信息呈现流程均符合预期；认知报告已经能够形成评分、趋势和关注项等结构化输出，表明系统并非仅输出简单提示文本，而是具备进一步解释和复核的基础。将这些结果与真实语音链路测量结果结合起来，可以看出本节功能已经覆盖“界面呈现、用户交互、异常降级、真实链路接入、性能边界识别”多个层面，因此比单纯的页面展示测试更能反映当前 AI 辅助能力的实际成熟度。

#### 3.2.2 测试结果综合分析及建议

建议后续将 live 语音服务测试纳入独立的 CI 任务或发布前检查清单，在显式开启 `VITEST_REAL_SERVICES=1` 的前提下执行，避免默认流水线长期停留在 Mock 级别；同时，应围绕语音克隆链路增加“预生成常用安抚语句”“缓存已合成短句”“执行前预热模型”等工程策略，以降低用户等待完整推理的概率。在前端交互层面，也建议为语音克隆增加更明确的进度提示、超时提示和兜底话术，避免用户将长时间等待误认为系统无响应。对于走失提醒功能，可继续补充“误报撤销”“连续多次触发去重”等异常场景验证；对于认知报告，则建议增加更大数据量下的渲染测试，以观察图表组件在历史数据增加后的可用性和稳定性。

#### 3.2.3 测试经验总结

AI 相关功能测试应同时覆盖模型接口、交互流程和降级处理，不能仅以“页面上是否出现内容”作为判断依据。本轮实践表明，将常规回归与独立 live 联调分离，是兼顾回归效率与真实服务验证的较优方案，因为这样既不会让日常开发被重型依赖拖慢，也不会让文档结论长期停留在 Mock 级别。

真实语音联调结果也进一步说明，语音服务能力评估必须同时考虑“是否可用”和“时延边界是否可接受”两个维度。对于识别和 TTS，这两个维度当前都已具备较好的基础；对于语音克隆，则需要在保持可用性的同时继续优化推理效率和前端等待体验。

本节 6 个接口用例全部通过，表明关键接口的功能行为和异常映射符合预期。`POST /api/elder/events` 在 100 次并发写入下保持 100% 成功率，且最终事件条数为 100，说明写入路径满足当前数据一致性要求。`GET /healthz` 和 `POST /api/elder/events` 的 P95 分别为 52.94ms 和 26.57ms，可以作为当前本地回环环境下的基线性能数据，但这些指标仅适用于本机单进程文件存储环境，不能直接外推至容器化或公网部署环境。事件写入平均时延较健康检查高 3.43ms，说明在当前数据规模下，落盘与事件追加开销处于可接受范围；`GET /healthz` 出现 82.05ms 的最大时延，则提示服务在冷启动或瞬时调度条件下存在轻微抖动，后续运行或部署应考虑预热策略。媒体上传与回读时延较低，也说明静态资源读写在当前环境下不会成为主要性能限制。

从性能分布看，当前数据服务在健康检查、老人档案查询、状态写入和事件写入等不同类型接口上均表现出较为稳定的响应特征，其中读请求和小请求体写请求耗时较低，事件落盘接口虽有额外开销，但未出现异常抖动或写入失败集中发生的情况。这一结果说明当前实现对于中等规模的本地演示型负载具备足够支撑能力。从并发正确性角度看，`POST /api/elder/events` 在 100 次请求后仍保持 100/100 落盘，具有较高参考价值，因为该接口不仅承担日志写入，还关系到异常轨迹记录、家属联动说明和后续状态追踪等多个业务环节。当前压测结果至少表明，在本地单进程文件存储环境下，写入路径不存在明显的竞争条件缺陷。

#### 4.2.2 测试结果综合分析及建议

建议在下一阶段增加更接近比赛部署形态的压测，例如 500 至 1000 请求量级、混合读写以及连续 30 分钟稳定性测试，以便进一步观察服务在更长时间窗口内的响应分布和数据完整性表现。若后续需要重点验证媒体上传、事件联动和家属通知等组合场景，则应提前进行冷启动与重复启动压测，以观察首次响应时间和资源预热特征；对于 `/api/elder/events`，还建议补充请求体大小变化、非法 JSON、超时重试和异常中断后的恢复行为等接口场景，从而进一步提升后端写入路径的稳健性。

#### 4.2.3 测试经验总结

接口与压力测试除关注响应时间外，更应关注数据完整性和错误处理行为。对本项目而言，事件记录、状态同步和通知链路均依赖后端数据一致性，因此“并发不丢失”比单纯追求更高吞吐量更符合业务验收重点。若性能数据好看，但并发写入后出现事件缺失、状态覆盖或错误恢复不充分，则测试结论在业务上并不成立。

本节经验进一步表明，在照护类系统中，性能指标必须结合业务性质解释。系统承担的是风险事件记录、家属提醒同步和老人状态管理等职责，因此“稳定写入、可回读、可复核”的价值往往高于“更高峰值吞吐”。后续若继续扩展压测规模，仍应优先关注成功率、数据完整性、顺序一致性和异常恢复行为，并将这些指标作为核心评价基线。

总体上，项目当前已形成老人端、家属端、数据服务、Bridge 与语音服务的基本联动闭环，测试结果能够支撑当前版本的功能验收与后续联调工作。

进一步综合各章节结果，可以形成以下判断。首先，项目内部主要模块之间的功能依赖关系已经经过分层验证，数据层、服务层、界面层和系统集成层之间不存在明显断点；其次，关键接口在本地真实服务环境下能够稳定处理读写请求，并在中等并发条件下维持数据完整；再次，外部语音能力已完成从 Mock 到 live 的补充验证，说明语音链路具备基础接入条件，只是在语音克隆推理效率方面仍需继续优化。

从测试文档的使用目的看，当前版本文档已经能够支持以下三类场景：一是用于说明项目当前版本的质量状态和验收边界；二是用于为后续联调、优化和问题复测提供结果基线；三是用于在后续补充截图、真机测试和长稳测试后继续扩展为更完整的阶段测试报告。后续如需进一步提高文档完备度，建议优先补充真实截图回填、移动端实机链路测试、长时稳定性测试和更高并发压力测试结果。
