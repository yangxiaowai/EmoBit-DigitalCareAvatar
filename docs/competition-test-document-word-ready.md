# 1 测试计划

## 1.1 测试策略与目标

### 测试策略

本项目属于面向老人照护场景的综合型智能系统，系统由老人端、家属端、Bridge 协议层、独立数据/媒体后端、语音识别服务、数字人交互能力和 Android 封装环境共同构成。测试工作不能只验证页面是否可打开或按钮是否可点击，而应从老人真实使用场景、家属协同管理场景、前后端数据闭环以及异常情况下的降级与恢复能力四个维度展开。

本轮测试采用“分层测试 + 场景驱动 + 接口校验 + 证据留痕”的组合策略。单元测试用于验证健康告警、异常轨迹判断、服药提醒、黄昏风险评估、人脸数据管理、认知评估、聊天兜底逻辑和独立数据后端等核心模块的规则正确性。功能测试围绕八个核心照护场景进行组织，重点验证老人端提示是否清晰、家属端配置是否生效、状态是否能够留痕并跨会话恢复。系统测试则进一步覆盖 Bridge 与 Data Backend 的接口契约、媒体上传与托管、前端 rehydrate、一致性、并发写入、语音服务降级、Android 网络访问和轻量性能表现。

为保证比赛答辩时测试结论可追溯、可证明，本轮测试要求对关键场景同时保留自动化测试结果、页面截图、接口回包、日志输出、状态数据文件或媒体文件证据。文档中除测试结果之外的说明性内容已预先写好，后续只需补充实际执行结果和证据材料即可。

### 测试目标

本轮测试的主要目标如下：

1. 验证老人身体健康异常、走失迷路、人脸识别、到家后播放时光相册、服药提醒、黄昏综合症、认知评估与健康日报、日常聊天八个核心场景能够按预期触发并形成完整处理链路。
2. 验证独立数据/媒体后端能够稳定提供状态持久化、事件追加、媒体上传与托管能力，并在同一 `elderId` 的并发写入下保持数据一致性。
3. 验证 Bridge 在保持原有对外协议结构不变的前提下，能够稳定承担状态读写、事件透传、上下文聚合、UI 命令与外发动作的桥接职责。
4. 验证前端能够在刷新、重进和切换页面后，从统一状态源完成关键模块的数据重建，避免仅依赖浏览器本地缓存。
5. 验证 Android 端在局域网环境下具备基本的访问、媒体加载、权限申请、语音降级和小屏交互能力。
6. 验证系统在比赛演示强度下具备足够的稳定性与可展示性，不因轻量并发、轮询、媒体访问或语音启停导致关键功能失效。

## 1.2 测试范围

### 1.2.1 测试广度

本轮测试覆盖的对象包括：

1. 老人端核心交互模块，包括健康提示、相册播放、服药提醒、黄昏风险提示、认知问答和数字人对话。
2. 家属端配置与管理模块，包括健康、认知、用药、照护计划、人脸相册和时光相册等配置入口。
3. Bridge 协议层，包括健康检查、状态接口、事件接口、上下文接口、UI 命令接口与外发动作接口。
4. 独立数据/媒体后端，包括老人状态持久化、事件追加、媒体上传、媒体托管和并发写入控制。
5. 语音服务相关模块，包括浏览器语音识别、FunASR、语音克隆和文字兜底模式。
6. Android WebView 运行环境，包括局域网访问、麦克风权限、媒体加载、页面交互与端点配置。

### 1.2.2 测试深度

本轮测试不仅覆盖功能是否存在，还覆盖以下深度维度：

1. 规则正确性：阈值判断、评分逻辑、状态机转换和事件生成是否符合设计预期。
2. 数据一致性：老人端、家属端、Bridge 与 Data Backend 回读数据是否一致。
3. 跨会话恢复能力：刷新、重进、重启后关键状态与媒体资源是否仍可恢复。
4. 接口规范性：请求路径、header、token、请求体和响应体格式是否稳定。
5. 降级与容错能力：语音服务失败、网络不可达、权限缺失等情况下系统是否给出合理兜底。
6. 可演示性：关键场景是否便于现场展示，信息反馈是否直观。

### 1.2.3 本轮不作为重点验收的内容

本轮测试不以互联网级超大规模并发能力作为重点，不对第三方大模型回复的具体自然语言表达做固定断言，也不将复杂生产部署容灾作为本次比赛文档的主要范围。对于语音和大模型相关能力，测试重点是接口可达、结构稳定、错误可降级和结果可展示。

## 6 数据存储与落盘设计（基于当前项目实现）

本章用于把“系统数据存到哪里、如何持久化、如何同步”按**当前仓库真实实现**写清楚，便于答辩时解释数据闭环与证据位置。

### 6.1 存储选型（现状）

本项目当前版本**未接入浏览器 IndexedDB** 作为主要持久化方案，也**未接入 MongoDB** 作为云端主存储。系统采用两类可追溯的落盘介质：

1. **Bridge 本地状态文件**：`openclaw/bridge/data/state.json`  
   用于保存演示态下的老人最新状态、历史事件、外发动作记录与 UI Commands 等信息，并对外提供统一的 Bridge HTTP 接口（如 `/healthz`、`/api/state`、`/api/events`、`/api/context/*`、`/api/ui/commands`）。
2. **独立 Data Backend（文件落盘型）**：`backend/data-server/data/`  
   用于提供更清晰的“状态文件 + 事件追加 + 媒体托管”能力，便于在测试与答辩阶段直接取证（状态 JSON、事件 NDJSON、上传文件目录）。

（插入图片：图一：Bridge 与 Data Backend 的数据落盘与接口分工示意图。）

### 6.2 Data Backend 落盘结构（现状）

Data Backend 在 `backend/data-server/data/` 下维护三类数据目录（以 `elderId=elder_demo` 为例）：

- **老人状态快照（JSON）**：`backend/data-server/data/elders/elder_demo.json`  
  保存老人端/家属端聚合后的最新状态快照（便于回读与重建）。
- **事件日志（NDJSON，Append-only）**：`backend/data-server/data/events/elder_demo.ndjson`  
  每行一个事件 JSON，采用追加写（append-only）方式记录交互、告警、提醒、识别、干预等事件，便于追溯与回放。
- **媒体文件托管（Uploads）**：`backend/data-server/data/uploads/<elderId>/<type>/<hash>.<ext>`  
  上传图片/音频等媒体以文件形式落盘，接口侧返回 `/media/...` 资源 URL 供前端加载。

（插入图片：图二：Data Backend 数据目录结构截图（elders/events/uploads）。）

### 6.3 Bridge 与 Data Backend 的同步与桥接职责（现状）

为解决“前端交互产生的数据如何留痕、如何被外部系统消费、如何回写 UI”三个问题，当前项目把 Bridge 设计为“协议桥 + 聚合层”，把 Data Backend 设计为“持久化 + 媒体托管层”：

1. **健康检查与可用性**  
   - Bridge：`GET /healthz`  
   - Data Backend：`GET /healthz`
2. **状态读写（示例路径）**  
   - Bridge：`GET /api/state`、`POST /api/state/:key`（面向演示与协议稳定性）  
   - Data Backend：`GET /api/elder`、`POST /api/elder/state/:key`（面向持久化与回读重建）
3. **事件追加（Append-only）**  
   - Bridge：`POST /api/events`（统一接收前端事件并写入 Bridge 状态文件，同时可转发到 OpenClaw）  
   - Data Backend：`POST /api/elder/events`（写入 NDJSON 事件日志并更新状态快照）
4. **媒体上传与访问**  
   - Data Backend：`POST /api/media/upload`、`GET /media/...`

（插入图片：图三：接口回包与落盘证据位置对照（示例：/healthz、/api/elder、/api/events）。）

### 6.4 一致性与并发写入策略（现状）

为降低演示态“并发写入导致丢写/覆盖”的风险，Data Backend 内部对同一 `elderId` 的写入采用**写队列串行化**并结合**临时文件原子替换**写入状态快照；事件日志采用 append-only 追加写，减少写入冲突。系统测试章节的“轻量并发写入”用例应以“无错误、无损坏、可回读一致”为验收准则。

（插入图片：图四：并发写入测试结果截图（无丢写/无损坏/可回读）。）
