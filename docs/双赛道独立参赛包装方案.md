# 忆护同行——面向老年智能陪伴、健康监测与风险预警的 AI 智慧照护系统 · 双赛道独立参赛包装方案

更新时间：2026-04-07

## 1. 结论先行

基于当前仓库能力、你提供的 2025 版模板，以及本地对代码和测试结果的核对，这个项目**适合拆成两个独立作品**：

1. `软件应用与开发类`：包装成一套**家庭协同智慧养老平台**
2. `人工智能应用`：包装成一套**失智老人认知风险评估与主动干预系统**

这两个方向都能建立在现有代码之上，但必须做到：

- 名称不同
- 目标用户不同
- 核心问题不同
- 功能边界不同
- 演示视频不同
- 设计文档叙事不同
- 源码提交目录不同

如果 05 赛道内部还要在“实践赛/挑战赛”二选一，**当前更适合报“人工智能实践赛”**，不建议直接报“人工智能挑战赛”。原因是当前项目更像“完整应用系统 + AI 核心能力”，而不是面向单一赛题的算法攻关型作品。

## 2. 两个赛道的评审重心差异

### 2.1 软件应用与开发类

从 `01-1/01-2/01-3` 的要求看，软件赛道重点看的是：

- 需求是否真实
- 系统是否完整
- 架构是否清晰
- 安装部署是否可复现
- 测试是否充分
- 是否有完整安装包、源码、文档、演示材料

也就是说，这条赛道更吃：

- 双端产品完成度
- 后端与数据闭环
- 工程质量
- 可演示性

### 2.2 人工智能应用

从 `05-1/05-2/05-3` 的要求看，AI 赛道重点看的是：

- 问题是否明确
- AI 技术方案是否成立
- 数据来源与样例是否清楚
- 测试指标是否能证明有效
- 系统是否体现 AI 的核心价值

也就是说，这条赛道更吃：

- AI 问题定义
- 数据与指标
- 模型/算法/规则融合方案
- 对比实验与效果证明

## 3. 当前项目的真实基础

当前仓库已经具备比较强的软件作品底座：

- 老人端 + 家属端双界面
- 独立数据后端
- Bridge 协议层
- 用药、游荡、黄昏、认知、时光相册、人脸识别、语音等场景模块
- 单元、功能、系统测试

本地核对结果：

- `npm run test:unit`：24/24 通过
- `npm run test:functional`：19 通过，3 跳过
- `npm run test:system`：14/14 通过
- `npm run build`：构建成功，但主包约 `1034.87 kB`

所以当前项目的特点很明确：

- **软件赛道底子强**
- **AI 赛道叙事有潜力，但证据链还不够强**

AI 赛道目前最大的短板不是“没有 AI”，而是：

- AI 能力还嵌在综合产品里，没有被抽成独立作品
- 数据集、标注、指标、对比实验还不够像 AI 赛道材料

## 4. 推荐拆分方案

## 4.1 作品 A：软件应用与开发类

### 建议作品名

`忆护同行——面向老年智能陪伴、健康监测与风险预警的 AI 智慧照护系统`

### 建议定位

面向居家养老场景的双端软件系统。老人端负责陪伴、提醒和交互，家属端负责配置、观察和干预，系统通过独立数据后端与桥接层保证状态留痕、通知联动和跨端一致。

### 核心卖点

- 老人端 + 家属端双端协同
- 异常提醒、服药、相册、认知、定位等功能完整
- 独立数据后端与 Bridge 支撑实际部署
- 具备测试、构建、运行链路，工程完整度高

### 这一作品里应该保留的模块

- `App.tsx`
- `components/Dashboard.tsx`
- `components/ElderlyApp.tsx`
- `backend/data-server/*`
- `openclaw/bridge/*`
- `services/medicationService.ts`
- `services/healthStateService.ts`
- `services/faceService.ts`
- `services/memoryService.ts`
- `services/locationAutomationService.ts`
- `services/openclawSyncService.ts`
- Android 封装与部署说明

### 这一作品里应该弱化的内容

不要把 AI 算法讲成这件作品的主角。这里的 AI 更适合作为“增强能力”出现，而不是答辩主轴。

应弱化：

- 复杂算法推导
- 模型训练叙事
- 数据集实验对比

应强化：

- 用户痛点
- 场景闭环
- 系统架构
- 可靠性测试
- 可安装、可运行、可演示

### 软件赛道的答辩主线

建议用“一个家庭的一天”来讲：

1. 家属录入老人资料、用药计划、家庭相册
2. 老人端进行日常陪伴、提醒、相册回忆
3. 异常行为/未按时服药/走失风险出现
4. 系统自动留痕并联动家属端
5. 家属查看状态、确认处理、继续配置

### 软件赛道还需要补强的地方

P0：

- 完整重写 README，去掉 AI Studio 默认说明
- 整理安装文档、部署文档、用户手册
- 准备正式演示账号、演示数据、截图与视频
- 把 Android/Web 部署流程固化

P1：

- 做代码分包，解决主包过大问题
- 清理演示态、模拟态和正式态混杂的问题
- 统一品牌名、图标、界面文案

P2：

- 增加更真实的社区/家庭使用场景数据
- 增加更完整的权限、日志与异常恢复说明

## 4.2 作品 B：人工智能应用

### 建议作品名

`护忆 CareMind：失智老人认知风险评估与主动干预系统`

### 建议定位

面向轻度认知障碍、阿尔茨海默症早期和高风险老人群体的 AI 应用系统。系统基于对话、轨迹、时间段、健康状态和记忆锚点进行多源分析，实现认知退化识别、游荡风险判断、黄昏综合征预警和个性化干预建议生成。

### 核心卖点

- 多模态风险评估，不只是聊天机器人
- 认知评估 + 游荡检测 + 黄昏风险 + 记忆干预形成闭环
- 本地优先、混合智能、适老化交互
- 可生成日报、风险报告和干预建议

### 这一作品里应该保留的模块

- `services/cognitiveService.ts`
- `services/wanderingService.ts`
- `services/sundowningService.ts`
- `services/memoryService.ts`
- `services/healthStateService.ts`
- `services/aiService.ts`
- `services/funasrService.ts`
- `services/voiceCloneService.ts`
- `services/speechService.ts`
- 认知报告、风险报告、干预看板相关界面

### 这一作品里应该弱化或剥离的模块

不要继续以“家属后台 + 老人端完整平台”来讲，否则会和软件赛道高度重合。

建议弱化：

- 家属后台的大量配置型功能
- 通用产品运营视角
- 过多安装部署细节
- 整个平台管理能力

建议聚焦：

- AI 问题定义
- 特征构建
- 规则/模型融合
- 评估指标
- 干预效果

### AI 赛道的答辩主线

建议用“早筛 + 预警 + 干预”来讲：

1. 老人日常对话被转写并提取时间/地点/重复问询等认知特征
2. 轨迹与时间段数据识别游荡和黄昏风险
3. 系统综合多源信号给出风险等级
4. AI 生成个性化干预方案
5. 记忆锚点、亲属声音、相册回忆等方式完成干预
6. 输出日报和趋势报告，支持家庭与机构复核

### AI 赛道必须补强的地方

P0：

- 构建一个**独立 AI 数据包**
- 明确训练集/验证集/测试集或规则验证集来源
- 给出可量化指标
- 做基线对比和消融实验

P1：

- 从“纯规则”升级为“规则 + 轻量模型/分类器/评分融合”
- 增加风险评分解释字段
- 输出标准化 AI 报告样例

P2：

- 做真实用户小样本试用或半结构化访谈
- 增加机构/社区医生视角的评估页面

### AI 赛道建议新增的实验指标

- 重复问询识别准确率
- 时间定向力问题识别准确率
- 空间定向力问题识别准确率
- 游荡识别 precision / recall / F1
- 黄昏风险分级准确率
- 干预建议采纳率
- 风险报告生成时延

### AI 赛道建议的技术升级

当前版本已有很强的规则和流程能力，但为了让“人工智能应用”更有说服力，建议增加至少一个可量化的 AI 核心：

1. 认知对话分类模型
   使用中文文本分类模型或 embedding + classifier，对“时间定向、空间定向、记忆重复、负向情绪”进行分类。

2. 风险融合模型
   将轨迹、时段、对话、生命体征等特征输入轻量模型，输出统一风险分值，而不是仅靠单一阈值判断。

3. 干预推荐策略
   让系统根据老人画像、风险类型、历史响应结果，自动选择“记忆相册/亲属语音/安抚话术/联系家属”等干预方式。

## 5. 如何把两个作品的重合度压低

这是最关键的部分。不能只是同一个系统改两个标题。

### 5.1 问题定义分开

软件赛道的问题定义：

- “居家养老缺少一套可落地的家庭协同照护软件系统”

AI 赛道的问题定义：

- “失智老人早期风险难以通过日常行为与对话被及时识别和主动干预”

### 5.2 用户对象分开

软件赛道：

- 老人
- 家属
- 社区照护者

AI 赛道：

- 高风险老人
- 认知障碍家庭
- 康复/养老机构评估人员

### 5.3 核心功能分开

软件赛道主功能：

- 双端协同
- 用药提醒
- 家庭相册
- 人脸识别
- 消息通知
- 数据持久化
- 部署运行

AI 赛道主功能：

- 认知评估
- 游荡识别
- 黄昏风险分析
- 多源风险融合
- 主动干预建议
- 趋势报告

### 5.4 界面分开

软件赛道界面建议保留：

- 家属后台
- 老人 App
- 运维/部署图

AI 赛道界面建议新做或重构为：

- 风险评估总览页
- 趋势分析页
- 干预建议页
- 个案报告页

最好不要直接复用同一套 Dashboard 截图。

### 5.5 视频脚本分开

软件赛道视频重点：

- 功能闭环
- 跨端联动
- 可部署可运行

AI 赛道视频重点：

- 输入什么数据
- 怎么分析
- 输出什么风险和建议
- 为什么比传统方案更有效

### 5.6 源码包分开

建议不要直接提交同一个仓库压缩包给两个赛道。

建议做法：

1. 抽公共层到 `shared-core`
2. 软件赛道保留 `platform-app`
3. AI 赛道保留 `ai-risk-app`
4. 两个包分别有各自 README、启动方式、截图、演示数据

如果时间不够做 monorepo 重构，至少也要做到：

- 两个分支
- 两套入口页面
- 两套说明文档
- 两套演示数据

## 7. 各赛道文档写法建议

## 7.1 软件应用与开发类文档写法

`01-3` 模板里重点写：

- 需求分析：养老场景、家属协同、现有产品不足
- 概要设计：双端 + 后端 + Bridge + 数据流
- 详细设计：模块、数据库、核心流程、交互界面
- 测试报告：功能、接口、系统、构建、稳定性
- 安装及使用：部署、启动、操作流程
- 项目总结：工程落地、推广价值

这里要把“AI”写成系统中的关键模块，而不是全文都在讲模型。

## 7.2 人工智能应用文档写法

`05-3` 模板里重点写：

- 问题来源：失智老人早筛和风险干预难
- 现有方案：量表、人工巡护、单一定位告警的不足
- 解决思路：对话 + 轨迹 + 健康 + 时段多源融合
- 技术方案：特征提取、分类/评分、干预策略
- 系统实现：数据采集、训练/规则融合、推理流程
- 测试分析：准确率、召回率、对比实验、案例分析

这里不要再把篇幅花在“完整软件平台工程”上。

## 8. 最现实的推进顺序

如果时间有限，建议按下面顺序推进：

### 第一优先级

- 先把软件赛道做成完整成品
- 因为当前代码和测试基础已经更接近软件赛道成品

### 第二优先级

- 再抽 AI 核心，包装成 AI 实践赛作品
- 重点补数据、指标、实验和独立界面

### 不建议的做法

- 不建议拿同一套 UI 和同一套 PPT 直接报两个赛道
- 不建议让两个作品都叫“AI 老年照护系统”
- 不建议 AI 赛道只讲功能，不讲数据与效果
- 不建议软件赛道把大量篇幅放在算法推导上

## 9. 最终建议

最优方案不是“一套系统投两个赛道”，而是：

- 以当前项目为母体
- 向外拆出两个叙事清晰、材料独立、演示独立的作品

推荐落点：

1. `软件应用与开发类`
   `忆护同行——面向老年智能陪伴、健康监测与风险预警的 AI 智慧照护系统`

2. `人工智能应用（优先实践赛）`
   `护忆 CareMind：失智老人认知风险评估与主动干预系统`

前者赢在工程完整度，后者赢在 AI 问题定义和智能价值。  
只要后续在“界面、文档、演示、代码包、指标体系”五个层面彻底拆开，这两个作品可以独立参赛，且重合度能控制在比较合理的范围内。
