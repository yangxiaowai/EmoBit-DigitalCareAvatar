# 软件赛道项目优化双人分工方案

更新时间：2026-04-07  
适用范围：`软件应用与开发类` 项目本体优化  
说明：本方案**只讨论项目改进方向、具体优化方案和双人分工**，**不包含** `01-2`、`01-3`、PPT、视频脚本等文档类写作任务。

## 1. 优化目标

当前项目已经具备以下基础：

- 双端界面：老人端 + 家属端
- 后端闭环：独立 Data Backend + Bridge
- 场景模块：用药、健康、轨迹、相册、认知、黄昏守护、语音
- 自动化验证：单元、功能、系统测试可执行
- 可构建：`npm run build` 已成功

但如果以 `软件应用与开发类` 参赛，项目本体还存在几个明显问题：

1. 对外界面仍有较重“研发演示版/模拟版”痕迹
2. 用户视角不够统一，老人端、家属端、控制台、调试状态混在一起
3. 运行依赖和演示依赖混杂，现场演示抗风险能力不够强
4. 构建体积偏大，功能组织方式不够收敛
5. 代码仓里存在旧文件、测试痕迹、临时文件，不利于做提交版源码包

因此软件赛道的项目优化目标应明确为：

- 把项目从“能跑的综合 demo”收敛成“完整、稳定、可展示的软件作品”
- 把评委可见部分做得更像产品，把评委看不见部分做得更像工程
- 保证截图、录屏、演示、安装、测试都能基于同一套稳定版本完成

## 2. 两人分工原则

建议两位成员固定承担如下角色：

- `成员A：前端产品化与演示体验负责人`
- `成员B：后端交付与工程收口负责人`

分工边界：

- A 负责“用户能看到和直接感知到的项目优化”
- B 负责“系统能稳定运行和可交付的项目优化”

这样划分的原因是：

- 软件赛道首先看“像不像一个完整的软件产品”，这部分主要由 A 收口
- 软件赛道也非常看“能不能跑、能不能测、能不能交付”，这部分主要由 B 收口

## 3. 总体优化方向

本项目的技术和产品优化，建议按六个方向推进：

1. `产品化收口`
2. `界面演示化收口`
3. `场景链路稳定化`
4. `运行配置标准化`
5. `构建与包体优化`
6. `提交版代码清理`

其中：

- 方向 1、2 以成员A为主
- 方向 3、4、5、6 以成员B为主
- 两人都需要共同验收“最终演示态”

## 4. 成员A分工：前端产品化与演示体验优化

## A-1 统一软件赛道的产品外观与命名

### 目标

把当前界面从“研发视角的模块集合”统一成“家庭协同智慧养老平台”的产品形态。

### 主要问题

当前代码中存在较多不利于软件赛道展示的对外表达：

- [App.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/App.tsx) 中直接出现“老人端 (App) / 家属端 (后台)”这类偏内部演示式命名
- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx) 中存在大量“模拟”“反控”“Bridge 未配置仅本地模拟”等开发态提示
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx) 中夹杂家属反控、预设场景、控制台操作提示等内容

### 优化方案

1. 统一命名体系：
   - 老人端改成更产品化的名称，如“长者关怀端”或“长者助手”
   - 家属端改成更产品化的名称，如“家庭照护中心”
   - 去掉“后台”“控制台”“反控”这类容易显得像演示工具的词
2. 统一首页标题、副标题、导航标签、卡片标题
3. 统一按钮文案：
   - “模拟”“预设”“调试”等替换为“场景演示”“一键触发”“演示模式”或在比赛模式下直接隐藏
4. 统一色彩和标题层级，使截图更稳定

### 涉及文件

- [App.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/App.tsx)
- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx)
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx)
- 如有必要，补充统一常量文件，例如 `config/branding.ts`

### 建议执行步骤

1. 列出当前所有用户可见名称和按钮文案
2. 输出一份“比赛版 UI 词典”
3. 逐页替换
4. 录一轮走查视频，看是否还残留开发态表述

### 验收标准

- 评委从首页进入后，不会看到明显的“开发测试口吻”
- 全系统只使用一套命名口径
- 所有截图区域的标题都适合直接进入 PPT 或文档

## A-2 把双端界面收敛成软件赛道主线

### 目标

把当前功能很多但略散的双端界面，收敛成软件赛道最强的主线流程。

### 主要问题

当前项目可展示能力很多，但软件赛道不需要“全都讲”。  
真正应该被强化的是：

- 家属录入与查看
- 老人端接受提醒与陪伴
- 异常事件触发
- 家属联动处理
- 状态留痕与记录

而不应在主界面上过度突出：

- 大量实验性入口
- AI 训练/推理感过强的模块
- 复杂的演示控制项

### 优化方案

1. 重新定义“家属端首页”的首屏重点：
   - 总览状态
   - 健康摘要
   - 定位与风险
   - 用药与相册入口
2. 重新定义“老人端首页”的首屏重点：
   - 时间天气
   - 数字人陪伴
   - 主消息区
   - 用药/相册/对话的明确入口
3. 将次级能力下沉到二级区域，不要与首屏争夺注意力
4. 演示时以“一天中的几件关键事”为线索，而不是功能堆砌

### 涉及文件

- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx)
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx)

### 建议执行步骤

1. 在 Dashboard 中确定 4 个一级展示重点
2. 将不属于一级重点的展示内容折叠或移动到二级区域
3. 在 ElderlyApp 中保留 3 个高频入口，其余通过次级入口进入
4. 录屏复测，看 30 秒内能否讲明白两个端分别做什么

### 验收标准

- 家属端首页看起来像“照护平台首页”
- 老人端首页看起来像“长者可使用的交互界面”
- 首屏中不再出现过多演示性操作控件

## A-3 隐藏或弱化开发态、模拟态、测试态元素

### 目标

保留演示能力，但不让评委一眼看出这是内部调试页面。

### 主要问题

当前项目里“模拟态”很重：

- [components/Sidebar.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Sidebar.tsx) 有“系统突发事件模拟”
- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx) 有大量“模拟轨迹/模拟黄昏/模拟健康数据”
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx) 里有大量“预设输入/已模拟XXX”

### 优化方案

建议增加一个“比赛展示模式”：

- 开启后，默认隐藏：
  - 开发日志
  - 调试提示
  - 明显的模拟按钮文案
  - 家属反控台中与评审无关的内部说明
- 仅保留对答辩必要的“一键场景切换能力”

### 技术建议

1. 在 [types.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/types.ts) 保留现有 `SimulationType`
2. 新增一个展示模式开关，例如：
   - `VITE_COMPETITION_MODE=true`
   - 或本地状态 `presentationMode`
3. 让组件根据该开关选择：
   - 比赛模式：隐藏技术说明
   - 开发模式：保留调试能力

### 涉及文件

- [App.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/App.tsx)
- [components/Sidebar.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Sidebar.tsx)
- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx)
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx)

### 验收标准

- 比赛模式下的界面干净、克制、适合录屏
- 开发模式下仍然能调试和演示
- 两种模式切换后不影响核心功能

## A-4 优化截图区与录屏区的页面表现

### 目标

把页面调整到“截出来就能用”的状态，减少后期二次修图压力。

### 优化方案

1. 所有关键页要有清晰的首屏视觉中心
2. 避免首屏出现过多滚动内容
3. 把关键数值、状态标签、流程按钮尽量放在同一屏
4. 保证浅色背景下文字可读性
5. 检查移动端模拟区和桌面端总览区是否存在截屏遮挡

### 重点页面

- 家属端总览页
- 健康页
- 定位页
- 相册/人脸页
- 老人端主界面
- 用药提醒弹层
- 游走告警弹层
- 时光相册界面

### 验收标准

- 不用滚动就能拿到主要截图
- 录屏时不会频繁出现布局挤压、内容跳动、弹层遮挡

## A-5 做“比赛演示态”的交互收口

### 目标

把系统常用演示链路做成稳定、短路径、低失误的交互。

### 优化方案

1. 将最常演示的 5 个场景固化：
   - 健康异常提醒
   - 异常轨迹/走失告警
   - 用药提醒
   - 相册回忆
   - 家属端查看与联动
2. 给这些场景提供统一入口，不要求评委现场复杂操作
3. 尽量让每个场景在 20 到 40 秒内完成闭环

### 验收标准

- 演示人员不需要临场在多个地方找按钮
- 每个场景都能稳定重复演示

## 5. 成员B分工：后端交付与工程收口优化

## B-1 标准化运行配置，减少现场演示风险

### 目标

把当前“能跑”升级为“可重复稳定运行”。

### 主要问题

当前运行依赖虽然已集中在：

- [utils/runtimeConfig.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/utils/runtimeConfig.ts)
- [openclaw/bridge/server.mjs](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/openclaw/bridge/server.mjs)
- [backend/data-server/server.mjs](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/backend/data-server/server.mjs)

但仍存在几个风险：

- 地址、端口、token 与模式切换缺少明确“比赛版默认值”
- 页面上会直接暴露“Bridge 未配置，仅本地模拟”这种技术性提示
- 不同服务的启停顺序和依赖关系没有做成工程化收口

### 优化方案

1. 增加“比赛演示环境”配置组：
   - 前端地址
   - Bridge 地址
   - Data Backend 地址
   - 是否开启展示模式
2. 将比赛模式和开发模式的默认值区分开
3. 为关键服务增加启动前检查和友好错误提示
4. 尽量减少现场依赖手工改 `.env`

### 涉及文件

- [utils/runtimeConfig.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/utils/runtimeConfig.ts)
- [openclaw/bridge/server.mjs](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/openclaw/bridge/server.mjs)
- [backend/data-server/server.mjs](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/backend/data-server/server.mjs)

### 建议执行步骤

1. 统一梳理当前所有运行时环境变量
2. 设计一个比赛版 `.env.example`
3. 给前端增加“关键服务未连接”的更友好 UI
4. 测试冷启动和重启流程

### 验收标准

- 换一台机器后，按固定配置可快速启动
- 服务未启动时，界面提示不显得像内部调试页

## B-2 清理“状态来源分散”的问题，强化可演示一致性

### 目标

减少页面展示依赖 `localStorage + 默认模拟数据 + 后端状态` 的混用状态。

### 主要问题

当前多个服务仍包含演示默认数据或本地存储逻辑，例如：

- [services/cognitiveService.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/services/cognitiveService.ts)
- [services/wanderingService.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/services/wanderingService.ts)
- [services/healthStateService.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/services/healthStateService.ts)
- [services/homeArrivalService.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/services/homeArrivalService.ts)

这会带来两个问题：

1. 刷新后数据来源可能不统一
2. 演示时不同页面可能呈现不同状态基准

### 优化方案

1. 为比赛演示统一一个“默认老人数据源”
2. 优先让比赛展示依赖后端状态或统一 seed 数据，而不是散落的本地模拟值
3. 给所有演示场景补一键重置功能，确保每次从同一起点开始

### 推荐思路

1. 以 [backend/data-server/data/elders/elder_demo.json](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/backend/data-server/data/elders/elder_demo.json) 为核心种子数据
2. 给各服务增加：
   - `loadFromBackendFirst`
   - `resetDemoState`
3. 让比赛模式下的首页默认从统一数据状态恢复

### 验收标准

- 同一场景刷新前后表现一致
- 两端看到的老人状态一致
- 演示前能一键恢复到初始状态

## B-3 重新组织演示能力，保留但不污染正式产品逻辑

### 目标

把演示和正式逻辑解耦，让系统既能演示，又不显得“全是模拟按钮”。

### 主要问题

当前演示能力深度嵌入组件：

- [App.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/App.tsx) 的 `handleSimulate`
- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx) 的多个模拟面板
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx) 的预设操作

### 优化方案

1. 把“演示触发能力”抽到单独的展示层模块
2. 在比赛模式下只保留必要入口
3. 业务模块仍保留真实处理逻辑，不因演示层而混乱

### 建议改造方式

1. 新建一个 `presentation` 或 `demo` 目录
2. 把：
   - 预设场景
   - 演示按钮
   - 一键触发逻辑
   从主业务组件中拆出去
3. 主业务组件只接收“发生了某个场景”的状态，而不关心它是用户触发还是演示触发

### 验收标准

- 主界面代码更清晰
- 比赛模式下可以极简触发场景
- 开发模式仍可保留完整调试入口

## B-4 优化构建体积与页面加载结构

### 目标

解决当前构建产物过大问题，提升软件赛道的工程完成度。

### 当前现状

本地构建结果显示：

- `dist/assets/index-*.js` 约 `1034.87 kB`

这会带来两个问题：

1. 软件赛道答辩时不利于体现工程质量
2. 如果现场环境一般，首屏加载稳定性会受影响

### 主要原因

从现有结构看，体积偏大的可能来源包括：

- [components/Dashboard.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/Dashboard.tsx) 单文件过大
- [components/ElderlyApp.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/ElderlyApp.tsx) 单文件过大
- `three`
- `recharts`
- 地图、语音、相册、3D 内容都在主包路径中

### 优化方案

1. 做路由级或视图级懒加载
   - 家属端懒加载
   - 老人端懒加载
2. 将重模块拆包：
   - 图表模块
   - 地图模块
   - 3D 模块
   - 语音模块
3. 根据需要在 [vite.config.ts](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/vite.config.ts) 中增加 `manualChunks`
4. 清理未使用依赖和历史组件

### 优先级建议

P0：

- `Dashboard` 与 `ElderlyApp` 分包
- 抽离大体积弹层和复杂子模块

P1：

- 地图、图表、3D 延迟加载

P2：

- 进一步检查资源文件体积

### 验收标准

- 主包体积明显下降
- 首屏加载稳定
- 不影响已有测试通过

## B-5 清理旧文件、临时文件和不应进入提交版的内容

### 目标

为后续源码提交和比赛分支做准备，清理仓库中容易引起误解的内容。

### 主要问题

当前仓库存在一些明显不适合作为提交版保留的内容：

- [components/old_app.tsx](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/old_app.tsx)
- [components/diff.txt](/Users/yangjun/Downloads/emobit---ai-elderly-care-system/components/diff.txt)
- `tmp-src/`
- `_tmp_report_extract/`
- `_tmp_report_extract.zip`
- `scripts/__pycache__/`
- 过多调试文档和临时产物

### 优化方案

1. 先分类：
   - 需要删除
   - 需要移出提交版
   - 需要保留但不进比赛源码包
2. 建议建立“比赛提交分支/目录”
3. 保证提交版代码仓不包含明显临时产物

### 验收标准

- 提交版源码目录清爽
- 不再出现“old”“tmp”“diff”等干扰评委判断的内容

## B-6 强化比赛前的工程回归与烟雾检查

### 目标

确保每一轮优化后，系统仍能稳定运行和演示。

### 优化方案

每次关键改动后都执行以下校验：

- `npm run test:unit`
- `npm run test:functional`
- `npm run test:system`
- `npm run build`

同时增加一个“比赛烟雾检查流程”：

1. 启动前端
2. 启动 Bridge
3. 启动 Data Backend
4. 打开老人端和家属端
5. 演示 5 个固定场景

### 验收标准

- 每次发布比赛版前都能完整跑通
- 优化不会悄悄破坏演示链路

## 6. 两人协同任务

## 7. 建议执行顺序

## 第一阶段：先把界面从“开发态”拉回“产品态”

## 8. 最终交付判断标准

当以下条件全部满足时，可以认为“软件赛道项目本体优化”基本完成：

- 首页和双端界面已经统一品牌与口径
- 比赛模式下不再出现明显调试/模拟痕迹
- 5 个核心场景能稳定演示
- 数据状态刷新前后基本一致
- 构建体积较当前版本有明确优化
- 仓库中旧文件、临时文件、无关文件已处理
- 每次改动后自动化测试和构建均可通过

## 9. 最终建议

这两个人的最佳协作方式不是“一个做前端，一个做后端”这么粗，而是：

- `A` 专门把项目做得更像软件作品
- `B` 专门把项目做得更像可交付工程

软件赛道最终评审看到的是产品，但给高分的前提一定是工程站得住。  
所以这份分工的核心逻辑是：

- A 解决“像不像作品”
- B 解决“能不能稳定交付”

两人最后一起解决“能不能在比赛现场稳定讲出来、演出来、跑出来”。
