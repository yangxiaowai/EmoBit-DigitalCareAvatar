## 1 测试计划

### 1.1 测试策略与目标

#### 测试策略

本项目采用分层递进的测试策略来覆盖“本地演示可复现”与“关键链路可验收”两类目标。首先在单元/模块层面，以 `services/`* 的核心业务逻辑为对象验证规则与边界条件是否成立，例如游走判定阈值是否会导致误报、黄昏风险评分在连续信号注入下是否具备平滑性与可解释性、用药提醒事件是否按时间与状态机生成、健康阈值告警是否能准确进入 warning/critical 等级。其次在功能层面，以四大场景（走失/游走、黄昏高风险、用药提醒、跌倒紧急）为端到端用例，验证“点击模拟按钮 → 同步入 Bridge → state.json 落盘 → 上下文可用 → 外发/回写（如启用）→ UI 展示”的闭环是否成立。最后在系统层面，把 Bridge、OpenClaw Gateway、插件与消息通道作为一个整体进行集成验证，重点关注外部依赖（飞书通道、OpenClaw CLI 调用）在真实运行时是否可达、可追踪、可回放，并通过性能与流量分析评估在答辩演示与未来扩展场景下的可承载性。

在优先级安排上，本项目坚持“风险优先”的执行原则，把最可能造成实际安全风险或最容易在演示中失败的链路提前覆盖。具体而言，走失/游走与跌倒紧急属于高优先级安全场景，必须保证事件产生、落盘、可被上下文检索、并且在需要时能触达家属；黄昏高风险属于持续性风险场景，验收时不仅要证明告警发生，更要证明干预计划的生成与状态流转清晰可解释；用药提醒属于日常高频场景，重点在于事件同步与升级判据（例如是否已通知、是否需要升级通知）可被系统准确记录，避免重复提醒或漏提醒。

为了让测试结果具备“可证明性”和“可复现性”，本项目从一开始就以可观测证据为核心组织测试记录。除界面截图外，所有关键结论都要求能够落到结构化证据上：命令行输出用于证明接口与通道调用结果（例如 curl 回包、OpenClaw CLI Sent 结果）；Bridge 落盘文件 `openclaw/bridge/data/state.json` 用于证明状态、事件、外发记录与 UI commands 的真实存在；前端界面中的“系统日志区/状态条”用于证明系统动作已被 UI 轮询回写并展示；飞书端消息截图用于证明外发确实触达目标。通过“命令输出 + state.json 证据块 + UI/飞书截图”的组合，可以把一次演示中的关键路径完整固化为可审阅、可追踪的验收材料。

#### 自动化测试体系（Vitest 分层落地）

为满足大赛模板对“单元测试—功能测试—系统测试”三层结构的要求，并降低人工回归成本，本项目在仓库内引入基于 Vitest 的自动化测试体系，形成可一键复现、可在持续集成中稳定执行的测试基线。自动化体系采用“三套配置文件分层隔离”的工程化方式落地：单元测试使用 `vitest.unit.config.ts`，功能测试使用 `vitest.functional.config.ts`，系统测试使用 `vitest.system.config.ts`；统一的测试运行环境与网络模拟由 `src/test/setup.ts` 启动，Bridge 端的接口行为通过 MSW（`src/test/msw/`*）在测试进程内模拟，从而使 OpenClaw 集成链路在无需真实 Bridge/飞书的情况下即可稳定复现与断言。

自动化体系的设计遵循“逻辑可验证、外部可模拟、证据可追溯”的原则：在单元测试层聚焦 `utils/`* 与 `services/`* 的确定性逻辑与协议守卫（例如 `utils/openclawMessageGuards.test.ts`），确保关键规则在边界条件下的输出可复核；在功能测试层以组件渲染与交互行为为中心，使用 JSDOM 环境与 Testing Library 对 UI 交互进行验证（例如 `components/Sidebar.test.tsx` 对关键模拟按钮的触发回调进行断言），从而把“按钮入口可触发”的验收要求固化为可重复执行的用例；在系统测试层采用“跨模块链路 + 协议契约”的方式，把 `openclawSyncService`（状态/事件同步）与 `openclawActionService`（outbound 动作）串联起来，在 MSW 模拟 Bridge 的前提下验证请求路径、header（token）、payload envelope 与关键时序（例如 `tests/system/openclaw-sync-and-action.msw.test.ts`），确保集成契约稳定不漂移。该体系使得测试记录不再仅依赖截图与手工复现，而能够输出可审阅的命令与用例执行结果，符合竞赛材料对严谨性与可复核性的要求。

为便于复现与审阅，本项目把三层测试的执行命令固化到 `package.json`：`npm run test:unit`、`npm run test:functional`、`npm run test:system` 可分别独立执行并输出清晰的分层结果；在提交材料或进行答辩演示前，可执行 `npm run test:run` 或 `npm run test:coverage` 对全量用例进行回归验证，从而确保“演示场景可复现”与“关键链路契约未漂移”两类目标同时满足。

#### 测试目标（发布质量等级）

本项目在发布与答辩演示阶段的质量目标可以概括为“关键链路可用、核心规则正确、运行稳定且可定位”。所谓关键链路可用，是指在不依赖人工干预的情况下，四大场景的模拟触发能够被系统正确接收与处理：前端触发后 Bridge 必须能写入事件与状态并提供对应上下文接口；当系统需要通知家属时，消息通道必须能够成功触达（以飞书实测 Sent 与端侧截图为准）；外发动作必须被 Bridge 记录为 `outboundEvents`，同时通过 `uiCommands` 回写，使得前端可以展示“系统已执行通知动作”，从而形成演示闭环。所谓核心规则正确，是指游走/走失判定、黄昏风险评分、健康阈值告警、用药提醒事件与去重/节流等关键逻辑在边界条件下不出现明显错误，且其输出具有可解释性，能够在 `state.json` 和 UI 中被一致地观察到。所谓稳定与可定位，是指 Bridge 的关键接口在本机演示环境下保持毫秒级响应并具备合理的资源占用，且当问题发生时能够通过 HTTP 返回码、CLI stderr、Bridge 日志与 `state.json` 记录快速定位故障点，从而保证演示过程可控、可复现、可修复。

### 1.2 测试范围

#### 测试广度（模块/功能）

本项目的测试范围从“业务能力覆盖面”和“系统实现边界”两方面进行界定。就业务能力而言，测试覆盖前端服务层的关键模块：`services/wanderingService.ts` 负责对位置点序列进行分析并产生游走/越界/走失相关事件；`services/sundowningService.ts` 负责黄昏风险评分、告警触发与干预计划生成；`services/medicationService.ts` 负责用药提醒、确认、延后等事件生成与日志记录；`services/healthStateService.ts` 负责健康阈值告警与头像状态映射；`services/cognitiveService.ts` 负责对话记录、认知评估条目生成与日报数据整理；`services/openclawSyncService.ts` 则负责把上述状态与事件以统一的协议同步到 Bridge，形成后续 OpenClaw 上下文与动作执行的输入基础。由于这些模块共同构成作品的核心业务闭环，因此测试不仅验证其独立逻辑是否正确，也验证它们在端到端链路中能否协同工作并产出可追踪证据。

就系统实现边界而言，测试还覆盖 `openclaw/bridge/server.mjs` 所提供的全部关键接口与落盘机制。Bridge 是作品的“中枢”组件：它接收前端同步的状态与事件，并在 `openclaw/bridge/data/state.json` 中持续维护老人侧的最新状态与历史事件；它通过 `/api/context/`* 将复杂状态聚合为 OpenClaw 可直接消费的上下文；它通过 `/api/outbound/*` 将 OpenClaw 的通知/外呼动作落地为真实的消息发送，并在成功或失败时写入可回放的 outbound 记录；它通过 `/api/ui/commands` 将 OpenClaw 的决策与动作结果回写给前端用于演示验收。因此测试范围也包含对 `/healthz`、`/api/state/*`、`/api/events`、`/api/context/*`、`/api/outbound/notify-guardians`、`/api/outbound/notify-elder`、`/api/outbound/voice-call`、`/api/ui/commands` 等接口的可用性、性能、以及返回数据结构一致性验证。

此外，考虑到作品采用 OpenClaw 作为编排与外部通道的执行层，测试范围同样包含 `openclaw/plugin` 插件对 `emobit_*` 工具的注册与调用链路，确保 OpenClaw 侧能够通过工具访问 Bridge 的上下文与执行外发动作。最后，为了满足答辩“可视化验收”的需求，测试也覆盖端到端演示 UI（`App.tsx`）：包括四大场景模拟按钮的触发逻辑、前端日志区/状态条对 Bridge UI commands 的轮询展示机制，以及在异常情况下前端是否能给出可理解的提示，从而保证测试过程可复现、可观察、可讲解。

#### 测试深度（重点链路）

在测试深度上，本项目不满足于“某个按钮点击后界面有变化”的浅层验证，而是以“完整闭环可追踪”为基本要求，将每个场景拆解为可验收的关键节点并逐一取证。标准链路从前端“模拟按钮”触发开始，随后由 `openclawSyncService` 将事件或状态同步到 Bridge；Bridge 接收到数据后必须完成落盘与状态更新，使得 `state.json` 中可以看到对应的事件记录与最新状态；在启用 OpenClaw 编排的情况下，Bridge 会在指定事件到达时通过 webhook 唤醒 OpenClaw agent，agent 再通过 `emobit_*` 工具拉取上下文并做出决策；当决策需要通知家属时，Bridge 会调用 OpenClaw CLI 通过飞书通道发送消息，并把外发结果写入 `outboundEvents`；同时 Bridge 会写入 `uiCommands` 的回写指令，前端通过轮询将这些指令映射成状态条与日志区展示。测试深度的核心意义在于：即便某个环节失败，仍可以通过“事件是否入库、上下文是否可用、外发是否触达、UI 是否回写”这几类证据快速定位故障点，保证测试结论可靠且具备工程解释力。

### 1.3 测试环境

#### 软件/运行环境（实际运行状态）

本次测试在本机开发与演示环境中完成，操作系统为 macOS（darwin 25.3.0）。前端采用 Vite 作为开发服务器运行，便于在答辩演示与测试执行时实时观察 UI 状态条与系统日志区的变化，并能够通过 HMR 快速迭代修复问题；本环境中前端访问地址为 `http://localhost:3000/`。Bridge 作为本作品的中枢服务以 Node.js 进程运行，通过 `npm run openclaw:bridge` 启动并监听 `4318` 端口，对外提供健康检查、状态/事件接入、上下文聚合、外发动作与 UI commands 回写等接口。OpenClaw Gateway 在本机 loopback 上运行并监听 `18789` 端口，用于承接 Bridge 的 webhook 唤醒与 agent 执行，并加载飞书通道插件以完成消息外发。消息通道选用 Feishu，测试时通过 OpenClaw CLI 的“Sent via Feishu”返回与飞书端实际收到消息截图共同确认通道可用性与可达性。

### 4.2 作品运行时数据流量及性能分析（必选）

本节以“运行时真实观测数据”为依据，对作品在本机演示环境中的数据流量、关键接口性能与进程资源占用进行分析。之所以必须包含该部分，是因为本作品涉及多端协作与消息通道集成：前端会持续把状态与事件同步到 Bridge，Bridge 既要为 OpenClaw 提供上下文聚合接口，又要在必要时通过 OpenClaw CLI 调用飞书通道对外发送通知。若不对数据流量与性能给出量化结果，就无法判断系统在答辩演示、以及未来扩展到真实多老人/多家属并发时的可承载范围与优化方向。

#### 4.2.1 数据流量口径与数据流（运行时链路说明）

作品运行时的主要数据流可以划分为三段：第一段是**前端 → Bridge** 的同步流量，前端通过 `openclawSyncService` 将老人档案、用药、健康、认知、游走、黄昏风险等状态与事件以 JSON 的形式发送到 Bridge（典型接口为 `/api/state/`* 与 `/api/events`），这段流量决定了 Bridge 落盘数据规模与 OpenClaw 上下文的实时性；第二段是**Bridge → 前端** 的回写流量，前端会轮询 `GET /api/ui/commands` 获取 OpenClaw 的动作回写（例如 outbound 记录、状态条更新等），这段流量决定了 UI 的可视化反馈延迟与页面负载；第三段是**Bridge → OpenClaw → 飞书** 的外发流量，Bridge 在需要通知家属时会执行 `openclaw message send` 调用飞书通道，这段流量的关键在于“消息能否触达、以及外发动作能否被系统记录并回写 UI”，同时也会带来一定的 CLI 调用开销。

为了便于评估，本节的流量测量采用“单次请求下载体积（size_download）”与“多次请求延迟分位数（p50/p95）”两类指标，并结合 `state.json` 的落盘体积、事件条目数量，给出在当前演示数据规模下的实际占用与增长趋势判断。

#### 4.2.2 Bridge 状态落盘体积与增长趋势（实测）

#### 4.2.3.1 轮询带宽占用估算（按竞赛报告口径给出公式与分钟/小时量级）

为了把“接口体积”转换为竞赛报告常用的“带宽/流量”口径，需要明确轮询周期与估算公式。本作品前端在运行时会通过 `setInterval(poll, 2000)` 轮询 Bridge 的 `GET /api/ui/commands` 接口，这意味着轮询周期 T 为 2 秒，对应每分钟请求次数约为 Q = 60 / T = 30 次。若以单次响应下载体积 S（单位：Byte）近似，则轮询产生的下行流量可按以下公式估算：

\text{流量（Byte/分钟）} \approx S \times \frac{60}{T}, \qquad
\text{流量（Byte/小时）} \approx S \times \frac{3600}{T}

在本次实测中，我们使用 `since=0` 拉取了全量 UI commands，得到单次响应体积 S \approx 35923\text{B}（约 35.9KB）。将该值代入 T=2\text{s} 的公式可得到“全量返回情况下”的保守上界：每分钟约 35923 \times 30 \approx 1{,}077{,}690\text{B}（约 1.03MB/min），每小时约 35923 \times 1800 \approx 64{,}661{,}400\text{B}（约 61.7MB/h）。需要说明的是，这个估算刻意偏保守，因为真实运行时前端会带上 `since=lastUiCommandTs` 参数，通常只返回“增量的新指令”；只有在页面长期离线后重新打开、或 `since` 被重置为 0 时，才会接近全量返回的体积。

为了让报告更贴近实际使用场景，我们同时给出一个“增量返回”的经验区间：当每次轮询仅返回 0–2 条新指令时，响应体积通常会落在 1KB–5KB 量级（具体取决于指令 payload 字段与是否包含 targets/message 文本）。在这种情况下，流量将约为 30KB/min–150KB/min，对应 1.8MB/h–9MB/h。该区间更符合答辩演示中“偶发触发场景、UI 低频回写”的真实负载特征。

为了便于 Word 模板呈现，下表给出不同体积假设下的分钟/小时流量估算结果（下行方向：Bridge→前端）：


| 轮询周期 T | 单次响应体积 S       | 估算请求数（次/分钟） | 估算流量（约/分钟） | 估算流量（约/小时） | 备注                  |
| ------ | -------------- | ----------- | ---------- | ---------- | ------------------- |
| 2s     | 35.9KB（35923B） | 30          | 1.03MB/min | 61.7MB/h   | `since=0` 全量拉取的保守上界 |
| 2s     | 5KB            | 30          | 150KB/min  | 9MB/h      | 增量返回偏大（含较长 message） |
| 2s     | 1KB            | 30          | 30KB/min   | 1.8MB/h    | 增量返回常见（少量指令）        |
| 2s     | 0KB            | 30          | ~0         | ~0         | 无新指令（但仍有 HTTP 开销）   |


需要进一步指出的是，上述估算仅统计了响应体积（`size_download`），没有把 HTTP header、TCP/IP 开销以及可能的压缩（gzip/brotli）纳入；在实际部署环境中如果启用了压缩，文本类 JSON 的有效传输体积通常会显著降低。对于竞赛答辩而言，采用“无压缩、按下载体积估算”的口径更保守、更容易解释，也更符合评委对容量上界的关注点。

#### 4.2.4 运行时进程资源占用（内存，实测）

为评估本作品在演示环境中的资源开销，我们对 Bridge 进程与 Vite 前端开发服务器进程的常驻内存（RSS）进行了采样。实测显示 Bridge 进程 RSS 约 28MB，Vite 进程 RSS 约 19MB，并且两者已持续运行 13 小时以上仍保持稳定，没有出现异常膨胀。这说明在当前演示规模下，Bridge 的状态维护、JSON 落盘与接口服务开销处于可控范围；前端开发服务器作为演示载体，其资源占用也较低。需要强调的是，Vite 的 RSS 不代表生产部署的静态站点资源开销，生产环境通常会进一步降低前端服务端占用；而 Bridge 在生产环境中仍需要根据“老人数量、事件密度、轮询频率、外发频率”进行容量规划。

#### 4.2.5 外发链路的性能与流量特征（飞书）

飞书通知链路属于“低频但高价值”的流量类型，其性能关注点与普通接口不同：关键不在于毫秒级延迟，而在于消息是否能够触达、以及系统是否能够在外发后留下可追溯证据。本作品采用 Bridge 调用 OpenClaw CLI 的方式发送飞书消息，因此每次外发都包含一次进程调用、插件加载、以及与飞书服务的交互。虽然该链路的耗时会显著高于本机 HTTP 接口，但对演示/告警业务而言属于合理成本；并且 Bridge 会把外发结果写入 `outboundEvents`，同时回写 `uiCommands` 的 `outbound.recorded`，使得在 UI 上可以展示“已通知家属”这一动作，形成闭环证据。该闭环已在 §3.2 中完成实测与取证。

此外需要注意：OpenClaw 在飞书插件侧存在“群组消息 allowlist 策略”的警告信息（`groupPolicy=allowlist` 且 allowFrom 为空时，群消息可能被静默丢弃）。本次测试目标为 `user:oc_...` 的单聊目标，因此消息可以正常发送并返回 Message ID；若后续需要对群聊发送，需要在 OpenClaw 配置中补齐 allowFrom 或将 groupPolicy 调整为 open，并在测试报告中增加群聊目标的专项验证用例。

#### 4.2.6 性能结论与优化建议

综合以上实测结果，本作品在“本机演示规模（单老人、有限事件量、轮询获取 UI commands）”下具有明确的性能优势：Bridge 关键读接口延迟稳定在 2ms 量级，`state.json` 体积约 406KB，内存占用约 28MB，能够支撑答辩演示所需的实时性与可视化闭环。若未来扩展到多老人、多家属、多设备并发场景，建议优先在以下方向进行工程化优化：一是对 `state.json` 做分片归档或改为轻量数据库存储，避免单文件无限增长；二是对前端轮询 `ui/commands` 采用“since 增量拉取 + 更长轮询间隔”降低重复数据下载量，或者改为 WebSocket 推送；三是对外发链路引入异步队列与重试机制，保证通道异常时可回放与可观测；四是对飞书群消息配置与授权策略做专项验收，避免“静默丢弃”影响告警可靠性。
