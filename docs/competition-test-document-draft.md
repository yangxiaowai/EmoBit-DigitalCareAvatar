## 1 测试计划

### 1.1 测试策略与目标

#### 测试策略

本项目属于面向老人照护场景的多模块协同系统，测试工作不能仅停留在“界面可打开、按钮可点击”的浅层验证，而应围绕核心使用场景、跨模块数据流和关键风险点展开。为保证测试结论既具有工程可信度，又便于在大赛答辩中展示，本轮测试采用“分层测试 + 场景驱动 + 证据留痕”的总体策略。

在测试层次上，本项目将测试划分为单元测试、功能测试、系统测试与性能/稳定性测试四个层面。单元测试用于验证各核心服务与规则模块的确定性逻辑是否正确，例如生命体征阈值判断、异常轨迹识别、黄昏风险评分、服药状态机、认知评估条目生成、数据后端并发写入与媒体存储等。功能测试用于验证老人端与家属端在真实业务场景中的表现是否符合预期，重点围绕健康异常提醒、走失迷路、人脸识别、时光相册、服药提醒、黄昏综合症、认知评估与日报、日常对话等八个高频核心场景展开。系统测试则重点验证前端、Bridge、独立数据后端、媒体托管、语音服务、Android 端网络配置之间的接口契约、集成链路和异常降级能力。性能与稳定性测试用于验证系统在答辩演示和日常演示强度下是否具备足够的响应速度、数据一致性与资源稳定性。

在测试组织方式上，本项目以“场景优先、风险优先”为原则。与老人安全、健康和家属告警直接相关的场景优先级最高，例如身体异常提醒、走失迷路、未按时吃药、黄昏综合症等；与情感陪伴和记忆唤起相关的场景，如人脸识别、时光相册、日常聊天、认知评估与健康日报，也作为核心功能进行重点覆盖。对于涉及外部依赖的能力，如 FunASR、语音克隆、媒体托管、Android 网络访问等，不仅要验证“可连通”，还要验证“失败后是否能降级、异常后是否能定位”。

为提高文档的可复核性，本轮测试建议统一保存以下证据：终端执行结果、自动化测试通过截图、浏览器/Android 页面截图、关键接口 JSON 回包、Bridge/Data Backend 落盘文件片段、媒体 URL 访问结果、日志输出和必要的录屏片段。这样可以保证每一个结论既有文字说明，也有可追溯的证据支撑。

#### 测试目标

本轮测试的总体目标，是验证项目在“老人核心照护功能可用、数据可持久化、媒体可托管、前后端链路可协同、异常场景可处理、Android 基础链路可运行”的前提下，是否具备进入答辩演示和作品验收的条件。具体目标包括以下几个方面：

1. 验证健康异常、走失迷路、人脸识别、到家相册、服药提醒、黄昏异常、认知评估与健康日报、日常聊天八个核心场景能够按预期触发、处理、展示和留痕。
2. 验证独立数据/媒体后端能够正确完成老人数据持久化、事件追加、媒体上传与托管，并保证同一 `elderId` 下的并发写入不会丢数据、不会被旧状态覆盖新状态。
3. 验证 Bridge 在保持原有对外协议不变的前提下，能够正确代理独立数据后端，并为前端和 OpenClaw 提供稳定的一致性数据来源。
4. 验证前端启动时能够从 Bridge 或后端读取并重建关键业务状态，保证刷新、重进、切换页面后，老人面板与家属端配置保持一致。
5. 验证 Android 端在局域网环境下具备基本的访问能力，包括后端连通、媒体加载、麦克风权限、语音识别降级与基础交互可用性。
6. 验证系统在演示级负载下具备基础性能与稳定性，不因并发写入、轮询、媒体访问或语音启停而出现明显错误、资源泄漏或关键功能失效。

### 1.2 测试范围

#### 1.2.1 功能范围

本轮测试覆盖的功能范围包括老人端、家属端、Bridge、独立数据/媒体后端、语音服务以及 Android 运行环境六个部分。

老人端测试范围重点包括：健康异常提醒展示、走失迷路告警、人脸识别与人物播报、到家后自动/手动打开时光相册、服药提醒弹层与确认、黄昏综合症风险卡片与安抚干预、认知评估与日报展示、老人和数字人的日常对话、语音不可用时的文字兜底交互。

家属端测试范围重点包括：老人基础资料与配置编辑、健康/认知/用药/照护计划查看与调整、人脸相册与时光相册条目管理、通过相应操作影响老人端状态、查看系统状态与接口健康情况、在需要时通过 Bridge 触发家属通知、界面在桌面与移动端布局下的可操作性。

独立数据/媒体后端测试范围重点包括：老人状态读取、分区状态写入、事件追加、媒体上传、媒体托管、默认老人数据结构补全、旧数据自举、并发写串行化、原子写与文件一致性。

Bridge 测试范围重点包括：健康检查、状态写入、状态读取、事件写入、上下文聚合、外发动作、UI 命令轮询、错误码与鉴权 header 的稳定性，以及对数据后端的代理一致性。

语音服务测试范围重点包括：浏览器语音识别可用性检测、FunASR WebSocket 可用性检测、语音不可用时的降级逻辑、语音录制启停的资源释放、语音服务失败时的用户提示是否明确。

Android 端测试范围重点包括：局域网访问 Bridge 和数据后端、媒体图片加载、麦克风权限申请、语音识别/文字兜底链路、端点可配置能力、小屏交互可操作性。

#### 1.2.2 非功能范围

本轮测试除功能正确性外，还覆盖以下非功能内容：

1. 接口正确性：请求路径、header、token、请求体与响应体结构、错误码与错误信息可定位性。
2. 数据一致性：刷新、重启、并发写入、跨页面跳转后状态是否保持一致。
3. 稳定性：轮询、媒体访问、语音启停、外部服务短暂失败后系统是否能恢复或给出明确降级提示。
4. 轻量性能：关键接口响应时间、状态落盘体积、媒体访问速度、基础资源占用。
5. 可演示性：UI 是否能清晰呈现异常、提醒、相册、日报、认知信息与操作反馈，便于比赛现场讲解。

#### 1.2.3 本轮不重点覆盖的内容

为控制测试范围，本轮测试不以“超大规模互联网高并发”作为重点，也不以第三方大模型生成内容的语义优劣作为主要验收对象。对于大模型与语音识别结果，不建议在测试中断言固定文案，而应重点验证返回结构、动作触发、降级路径和结果是否被正确展示、记录与持久化。

#### 1.3.3 测试数据准备

#### 1.3.5 风险与约束

本轮测试存在以下客观约束，需要在文档中提前说明：

1. 语音识别和语音克隆存在外部依赖，结果可能受服务部署状态、网络连通性和权限限制影响。
